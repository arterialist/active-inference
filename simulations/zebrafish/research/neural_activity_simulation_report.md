# Zebrafish Neural Activity Simulation Report

Date: 2026-05-13

## Runtime Findings

The current zebrafish runtime does build and tick a PAULA network:

- `connectome.py` creates 104 named neurons and 145 directed chemical edges.
- `neuron_mapping.py` injects sensory inputs into matching PAULA sensory neurons.
- `NeuronNetwork.run_tick()` advances PAULA state once per physics tick by default.
- The lab snapshots expose compact PAULA state as `neural.names`, `neural.s`,
  `neural.r`, and `neural.fired`.

Measured after the current patch in a 2500-step smoke run:

- neuron count: 104
- maximum `S` observed: 0.7247
- mean `S` range: 0.0002 to 0.0356
- maximum simultaneously fired neurons: 1
- mean fired neurons per tick: 0.0188

The activity is therefore still mostly graded/subthreshold. The visible neural
panel is not showing a rich whole-brain calcium-like trace; it is showing the
current PAULA state variables for a small scaffold circuit.

## What Actually Drives Behavior

The current locomotion loop is not yet a pure connectome-to-muscle controller.
The behavioral state in `ZebrafishNervousSystem` computes:

- `swim_drive`
- `turn_bias`
- `pitch_bias`
- bout/coast timing
- startle timing

Those values are computed directly from encoded sensory channels such as retina,
optic flow, lateral line, wall, depth, thermal, olfactory, and startle inputs.
The tail motor outputs are then generated by `_decode_tail_motors()` as a
traveling tail wave modulated by drive, turn, and pitch.

PAULA is active, but its firing/spiking output is not yet the necessary source
of motor control. The code nudges selected PAULA state variables so the neural
viewer reflects sensory/behavior/motor state, but movement would still work if
the PAULA connectome were replaced by a simpler state object.

## Calcium And Public Activity Data Usage

The public calcium/voltage imaging resources are currently used as source
corpus and design guidance, not as numerical runtime data.

Recorded in the corpus but not loaded by the simulation:

- Ahrens et al. 2012/2013: brain-wide calcium imaging and whole-brain
  light-sheet recordings.
- Portugues et al. 2014: whole-brain activity maps for visuomotor behavior.
- Dunn et al. 2016: brain-wide activity during exploratory locomotion and ARTR
  dynamics.
- Randlett et al. 2015 / Z-Brain: atlas and MAP-mapping region activity.
- Sy et al. 2023 / FishChip: chemosensory behavior and brain-wide
  representation.
- DANDI thermal, OMR, voltage, and calcium datasets listed in
  `source_catalog.md` and `source_manifest.json`.

Not implemented yet:

- no calcium trace files are loaded at runtime;
- no atlas-to-neuron mapping is applied;
- no fitted time constants, covariance, region priors, or activity templates
  are used in PAULA parameters;
- no comparison metric validates simulated `S` against GCaMP, CaMPARI, pERK, or
  voltage-imaging data.

## Is The PAULA Connectome Necessary Right Now?

For the current zebrafish demo behavior: no, not yet. It is useful as a
scaffold for the lab, the connectome viewer, PAULA parameter inspection, and
future replacement of the hand-coded controller. But it is not yet causally
necessary for swimming.

For the intended digital twin: yes. The next necessary step is to make the
PAULA circuit causally own the behavioral variables:

1. sensory inputs should drive PAULA regions;
2. PAULA sensory/interneuron/motor state should determine swim gate, turn, and
   pitch;
3. public calcium/voltage datasets should constrain neural time constants and
   region-level activity;
4. the reduced 104-neuron scaffold should be replaced or augmented with the
   strongest available public connectome exports.

Until that is done, the zebrafish simulation is a MuJoCo anatomical/body and
environment parity implementation with a PAULA neural scaffold, not a full
calcium-grounded connectomic digital twin.
